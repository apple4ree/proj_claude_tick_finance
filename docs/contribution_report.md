## 1. 연구 배경과 현재 문헌의 공백

### 1.1 최근 LLM-트레이딩 논문의 흐름 (2026년 2~4월)

2026년 들어 LLM이 트레이딩 전략 코드를 생성하는 연구가 폭증했다. 대표적으로 다음 10편을 docs/ 폴더에 수집해 정독했다.

| 논문 | 소속 | 주요 주장 |
|---|---|---|
| FactorMiner | 칭화대 | 자가진화 에이전트가 A주 팩터 110개 발견, IC 8.25% |
| AlphaForgeBench | NTU, 홍콩과기대 (KDD'26) | 프론티어 LLM 6종이 코드 생성률 96% 이상, 샤프 0.6~0.9 |
| Beyond Prompting | 베이징대 | 에이전트 AI가 미국 주식 long-short로 연환산 샤프 2.75 주장 |
| AlphaLogics | 선전대 | 시장 논리 기반 팩터 마이닝, IR 1.27~1.53 |
| TradeFM | JP모건 | 틱 데이터 기반 524M 파라미터 foundation model (트레이딩 전략 아님) |
| TradingAgents | Tauric Research | 다중 에이전트 트레이딩 프레임워크 |

**공통점 (문제의 핵심)**: 이 10편 중 9편이 **하루봉 또는 10분봉** 사용. **1편(TradeFM)만 틱 데이터**, 그러나 트레이딩 전략이 아니라 시장 시뮬레이션용 foundation model.

**즉 LLM × 틱-레벨 트레이딩 전략 생성 조합은 literature에서 blue ocean이다.**

### 1.2 왜 모두 하루봉만 쓰는가

세 가지 이유가 있다:

1. **LLM 훈련 데이터의 편향**: 교과서, Fama-French 학술 논문, Qlib 튜토리얼, Kaggle 대회 대부분이 하루봉 기반. LLM이 하루봉 지식은 풍부하나 마이크로구조 지식은 빈약.
2. **수수료-엣지 비율**: 하루봉에서는 엣지가 수십~수백 베이시스 포인트라 수수료(10~30bps)를 쉽게 넘김. 틱에서는 엣지가 단일 베이시스 포인트 수준이라 수수료가 바로 먹힘.
3. **데이터 접근성**: 하루봉 OHLCV는 무료 공개, 틱-레벨 주문서 데이터는 수집 비용이 큼.

### 1.3 현재 연구가 놓친 질문

기존 10편은 모두 "하루봉에서 LLM이 얼마나 잘 하는가"만 측정. 아무도 **"시간 단위를 틱까지 좁히면 LLM의 행동이 어떻게 달라지는가?"** 를 묻지 않았다.

이 질문을 물어봐야 하는 세 가지 이유:

1. **실전에서 중요**: 실제 거래는 틱 단위에서 체결됨. 하루봉 결과가 틱 실행으로 이어진다는 보장이 없음.
2. **이론적으로 흥미로움**: LLM의 도메인 지식이 시간 단위에 따라 비대칭적으로 분포한다면, 그 분포를 측정하는 것 자체가 AI의 한계에 대한 근본 통찰.
3. **실용적 경고**: 하루봉에서 잘 된 LLM 전략을 기계적으로 틱에 적용하면 구조적으로 실패한다는 경고.

## 2. 기여점 1 — 측정 방법론: 알파와 충실도의 분해

### 2.1 문제 정의

LLM이 작성한 트레이딩 전략의 수익 R은 두 가지 성분의 합이다.

$$ R = \underbrace{R_\alpha}_{\text{진짜 신호에서 온 수익}} + \underbrace{R_\text{bug}}_{\text{명세 위반에서 온 수익}} $$

기존 문헌은 $R$ 전체만 보고한다. 하지만 이 둘은 매우 다른 의미를 가진다.

- $R_\alpha$ 가 크다 → 정말로 LLM이 edge를 발견한 것
- $R_\text{bug}$ 가 크다 → LLM이 자기가 선언한 명세를 어긴 덕분에 우연히 수익이 나거나 손실이 커진 것

**두 논문이 같은 $R$ 을 보고해도 내용은 완전히 다를 수 있다.**

### 2.2 기존 연구의 한계

기존 LLM 코드 생성 평가는 세 가지 방식으로 성능을 측정한다.

| 방식 | 측정하는 것 | 한계 |
|---|---|---|
| pass@k (HumanEval) | 코드가 테스트를 통과하는가 | 트레이딩 도메인엔 unit test oracle이 없음 |
| run-to-run variance (AlphaForgeBench) | 같은 입력에 같은 출력이 나오는가 | 결정적으로 출력해도 명세 위반이면 의미 없음 |
| Sharpe / IR (대부분의 논문) | 위험조정 수익이 얼마인가 | $R_\alpha$ 와 $R_\text{bug}$ 를 섞어서 보고 |

아무도 $R_\alpha$ 와 $R_\text{bug}$ 를 분리해서 보고하지 않는다. **이 분리가 우리의 방법론 기여.**

### 2.3 우리의 접근 (dual-mode backtest)

같은 전략 코드를 두 가지 엔진 모드에서 실행한다.

- **일반 모드 (normal)**: 전략 코드가 시키는 대로 주문, 명세 위반이 있어도 체결됨
- **엄격 모드 (strict)**: 엔진이 LLM 자신이 선언한 명세를 **강제 집행**. 위반하려는 주문은 REJECT, 위반한 포지션은 FORCE_SELL

$R_\alpha = R_\text{strict}$, $R_\text{bug} = R_\text{normal} - R_\text{strict}$

$\text{clean\_pct} = (R_\text{strict} / R_\text{normal}) \times 100\%$

이게 **"LLM이 자기 명세를 얼마나 잘 지켰는가"를 수익 기여도로 환산한 지표**.

### 2.4 구체 예시 — strat_20260417_0001

LLM이 생성한 멀티-심볼 전략이 다음과 같이 명세에 작성.

```yaml
max_position_per_symbol: 3   # 한 종목당 최대 3주
```

그런데 전략 코드 자체는 이 제한을 제대로 강제하지 않았다. 지연 시간 동안 여러 주문이 경쟁해서 포지션이 11주까지 쌓이는 경쟁 조건이 있음.

**결과**:
- 일반 모드: 수익 -6,291원 (손실)
- 엄격 모드: 수익 -3,770원 (엔진이 11주 시점을 3주로 묶어줬을 때)
- 버그 기여: -2,521원 (손실의 40.1%가 명세 위반 때문)

**기존 리뷰어의 반응**: "이 전략 수익 안 나네요, 신호가 나쁘네요" 로 끝.
**우리 프레임워크 적용 후**: "이 전략 손실의 60%만 신호 문제이고, 40%는 에이전트가 자기 명세를 못 지킨 탓입니다. 신호는 덜 나쁠 수 있습니다."

이건 완전히 다른 해석.

### 2.5 왜 novel한가 (교수님 관점 설득)

교수님 질문 예상: *"기존 pass@k로도 비슷한 걸 측정할 수 있지 않나?"*

답: **불가능.** 이유:
- pass@k는 unit test가 있어야 동작. 트레이딩은 unit test가 없음 (수익이 양수면 통과? 그럼 단순 buy-hold도 통과)
- pass@k는 **이진 판정** (통과/실패). 우리 메트릭은 **연속값** (수익 기여도 0~100%)
- pass@k는 **외부 오라클** 필요. 우리는 **LLM 자신의 명세**가 오라클 역할

이게 첫 번째 novel point. **"unit test가 없는 도메인에서 spec을 oracle 삼아 fidelity를 연속 측정하는 방법"**.

교수님 질문 예상 2: *"그런데 이 메트릭이 실제로 유용한가?"*

답: 3개 전략에서 **메트릭 붕괴 유형을 우리가 역으로 발견**했다.

- 유형 A (strat_0003): 전략 코드에 "5틱 안 움직이면 손절 안 함" 같은 **추가 안전장치**가 있는데, 명세에는 안 써있음. 엄격 모드가 이 장치를 모르고 일찍 청산 → clean_pct = -1150% (무의미한 값)
- 유형 B (strat_0005): 명세 작성자와 검사기가 "초"를 다른 기준으로 해석 → 엄격 모드가 모든 진입을 차단 → clean_pct = 0%
- 유형 C (strat_0004, _0006): 명세 위반 없음, 그냥 신호가 나쁨 → clean_pct = 100% (정확한 판정)

**이 3가지 붕괴 유형을 발견하고 각각에 조건부 사용 규칙을 제시한 것**이 방법론 기여의 완성.

### 2.6 요약 — 기여점 1의 두 가지 새로움

1. **LLM 자신의 명세를 오라클 삼는 counterfactual 측정** — 이 아이디어 자체가 문헌에 없음
2. **메트릭 붕괴 3유형 분류와 조건부 사용 규칙** — 실무자에게 실용적 가이드

## 3. 기여점 2 — 시간 단위 임계값 개념

### 3.1 문제 정의

"LLM이 트레이딩 전략을 얼마나 잘 짜는가?"는 **시간 단위(horizon)에 매우 민감한 질문**이다. 같은 LLM이 같은 작업을 수행해도, 데이터 aggregation 단위에 따라 결과가 근본적으로 달라진다.

구체적으로 우리는 두 극단을 관찰했다.

| 시간 단위 | 전략 수 | 수익 전략 | 샤프 중앙값 | 최고 OOS 샤프 |
|---|---|---|---|---|
| 틱 (KRX) | 6 | 0 | -0.85 | -0.04 |
| 하루봉 (Binance) | 5 | 3 OOS | +0.17 OOS | **+1.12** |

같은 Claude Sonnet 4.6, 같은 에이전트 파이프라인, 같은 측정 레이어. 다른 것은 **시간 단위뿐**.

### 3.2 기존 연구의 한계

기존 10편의 LLM-트레이딩 논문은 전부 **단일 horizon**에서만 측정한다.

- 9편: 하루봉 또는 10분봉
- 1편: 틱 (하지만 전략 생성이 아니라 시뮬레이터)

**아무도 horizon을 바꿔가며 같은 LLM을 재평가하지 않았다.** 따라서:

- 하루봉 논문 9편의 결과가 **다른 시간 단위에 일반화되는지 아무도 모름**
- "샤프 2.75 나옵니다" 주장이 5분봉에서도 유지되는지 알 수 없음
- 실전 체결은 틱 단위 → 하루봉 실적의 **외부 타당성(external validity)이 검증 안 됨**

### 3.3 우리의 접근 — 시간 단위 임계값

**가설**: 각 (도메인, 수수료) 조합마다 **"이 시간 단위 이하에서는 LLM이 아무리 잘 만들어도 구조적으로 수익 불가능"** 한 임계값 $h^*$ 가 존재한다.

직관:
- 엣지 크기 $E(h)$ 는 horizon에 거의 선형 증가 (긴 구간일수록 큰 수익)
- 수수료 $C$ 는 거래 횟수에 비례, 긴 구간일수록 거래 줄어 총 수수료 감소
- **$E(h) > C(h)$ 이 되는 가장 작은 $h$ 가 임계값 $h^*$**

현재 증거:
- 크립토 10bps, 틱: $E < C$ → 수익 불가
- 크립토 10bps, 하루봉: $E > C$ → 수익 가능 (일부 전략에서)

**실험 1에서 할 일**: 6개 horizon (틱 / 1분 / 5분 / 15분 / 1시간 / 하루) 에서 같은 전략 돌려서 $h^*$ 를 실제로 측정.

### 3.4 구체 예시 — 틱 실패의 메커니즘

앞서 만든 틱 전략 strat_20260417_0004 (010140 spread-bps 전략):
- 명세에서 스톱로스를 **브리프 추천값 3 bps로 설정**
- 그러나 010140의 실제 틱 사이즈는 **약 16 bps**
- 즉 **3 bps는 1틱보다 작아 물리적으로 도달 불가능한 값**

LLM은 하루봉 지식으로는 "3 bps 스톱로스" 가 합리적이라 판단. 하지만 마이크로구조 지식 부족으로 이 값이 호가 단위보다 작다는 걸 모름.

→ 실행 결과: 8개 거래 모두 스톱로스, 승률 12.5%, 수익 -1.8%

**이 실패는 하루봉에서는 절대 발생 안 함** (하루봉 변동폭은 수백~수천 bps이므로 3 bps 같은 극단적 값은 LLM이 애초에 안 씀). 순수하게 horizon-specific 실패.

### 3.5 왜 novel한가

교수님 질문 예상: *"horizon 차이가 크다는 건 당연하지 않은가?"*

답: **"당연"해 보이지만 아무도 측정 안 했음.** 실증 연구의 본질은 "당연한 것을 숫자로 보이는 것".

- "모든 LLM 연구가 특정 horizon에 편향되어 있다"는 비판을 **단일 커브로 증명** 가능
- 임계값 $h^*$ 가 수수료에 따라 어떻게 이동하는지 $\partial h^* / \partial C$ 같은 정량 관계 유도 가능
- 실무자에게 **"당신 fee에서는 최소 15분봉 이상 써야 합니다"** 같은 구체적 가이드 제공

교수님 질문 예상 2: *"이게 방법론 논문인가 실증 논문인가?"*

답: **둘 다.**
- 방법론: 측정 레이어가 horizon-portable하다 (동일 프레임워크가 틱부터 하루봉까지 작동)
- 실증: 임계값이 실재하고, 그 위치를 측정

이중 기여라서 오히려 강함.

### 3.6 요약 — 기여점 2의 두 가지 새로움

1. **horizon sweep 자체가 처음** — 이 질문을 처음 묻는다는 게 주요 기여
2. **임계값 $h^*$ 라는 새 개념** — 수수료를 주면 수익 가능 최소 horizon이 결정된다는 물리화

## 4. 기여점 3 — 틱 레벨 실패 4가지 유형 분류

### 4.1 문제 정의

"틱에서는 LLM이 실패한다"만으로는 학술적 기여가 약함. **어떻게 실패하는지** 체계적으로 분류해야 실무자에게 처방 가능.

우리는 6개 틱 전략의 실제 실패를 분석해 **4가지 구조적으로 다른 실패 유형**을 추출했다.

### 4.2 4가지 유형 상세

#### 유형 1 — 명세-코드 간 해석 불일치 (Spec-implementation drift)

**정의**: 에이전트 A가 파라미터에 넣은 의미와 에이전트 B가 그 파라미터를 해석한 의미가 다름.

**구체 예**: strat_20260417_0005 (034020 spread 전략)
- 명세 작성자가 `entry_end_time_seconds: 14400` 을 **"장 시작 후 14400초 = 13:00 KST"** 로 의도
- 전략 구현자도 같은 해석으로 구현
- 그러나 우리 검사기는 **"자정부터 14400초 = 새벽 04:00"** 로 해석
- 결과: 매 거래가 "시간 제한 위반"으로 잘못 플래그 → 18건의 위조 위반

**이 실패의 신호**: 검사기가 유난히 많은 "위반"을 쏟아내는 반면, 실제 시간은 문제없어 보임. 이게 drift의 signature.

**감지 가능?**: **예, 우리 프레임워크로 감지됨** (18 위반이 한 전략에 집중).

**치료**: 스펙 컨벤션 통일 체크리스트.

#### 유형 2 — 마이크로구조 도메인 지식 부족 (Knowledge gaps)

**정의**: LLM이 틱 사이즈, 스프레드 구조, 최소 가격 단위 같은 **마이크로구조 규칙을 잘못 알고 있음**.

**구체 예 1**: strat_20260417_0004 (010140 spread 전략)
- LLM이 "KRX 10,000~50,000원 가격대 틱은 100원" 이라 주장
- 실제로는 **50원**
- 결과: "1틱 스톱" 이라 생각한 값이 실제로는 "2틱 스톱" 이 되어 손실이 2배

**구체 예 2**: 거의 모든 틱 전략에서
- 브리프의 stop_loss_bps = 3 을 그대로 받음
- 그런데 16 bps/틱인 종목에서 3 bps 는 **0.2틱**, 물리적으로 구현 불가
- LLM이 이를 감지하고 200~1000% 상향 조정 — 일종의 땜빵

**감지 가능?**: 부분적. 실행 후 sl_overshoot 위반으로 잡히지만, **사전 정적 분석으로는 못 잡음**.

**치료**: `sub_tick_sl_declaration` 같은 **새 invariant 유형** 추가. 스펙 로딩 시점에 "stop_loss_bps < 1-tick bps of target symbol" 정적 체크.

#### 유형 3 — 에이전트 간 정보 손실 (Handoff decay)

**정의**: 상류 에이전트가 출력한 필수 필드를 하류 에이전트가 **말없이 누락**.

**구체 예**: 
- alpha-designer는 `signal_brief_rank: 2` 반환
- spec-writer가 idea.json 생성할 때 이 필드가 **사라짐**
- 명시적으로 "이 필드 보존하시오" 지시 없으면 100% 누락

**정량 측정**: 
- 명시적 지시 없음 (n=1): 필드 유지율 **11%**
- 명시적 지시 있음 (n=5): 필드 유지율 **96%**

**감지 가능?**: 예, handoff_audit.json 으로.

**치료**: 
- 단기: prompt에 "이 필드를 반드시 verbatim 유지" 명시
- 장기: 구조화된 출력 강제 (Pydantic 등)

#### 유형 4 — 검사기 맹점 (Invariant taxonomy blindspots)

**정의**: LLM 코드가 명백히 틀렸는데 우리 7유형 검사기가 **검출 못함**.

**구체 예 1**: strat_20260417_0005 의 OBI 게이트 누수
- 코드: "OBI > 0.5 일 때만 매수"
- 실제 체결: 18건 중 8건이 OBI < 0.5 상태에서 체결 (2건은 음수)
- 원인: 전략 코드가 `update_state()` 전에 OBI를 읽어서 stale value 사용
- 우리 검사기는 "시간 게이트" 는 체크하지만 "신호 조건 게이트" 는 체크 안 함

**구체 예 2**: 지정가→시장가 전환 불리한 체결
- 전략이 "현재 매수호가에 지정가 매수" 주문
- 지연 시간 5ms 동안 호가가 떨어짐
- 주문이 "marketable" 상태가 되어 매도호가에서 체결 (불리)
- 우리 검사기는 이 패턴을 인식 못함

**감지 가능?**: **불가능 (현재).** 이게 맹점이라는 이름의 이유.

**치료**: 새 invariant 유형 제안
- `entry_signal_gate_breach`: 체결 시점에 의사결정 조건 재검증
- `limit_marketable_crossover`: 지연 시간 동안 주문 상태 변화 감지
- `sub_tick_sl_declaration`: 정적 분석으로 불가능한 임계값 탐지

### 4.4 실험 3과의 연결

실험 3 (prompt intervention sweep)에서 각 intervention이 어느 유형을 줄이는지 측정.

| Intervention | 목표 유형 | 가설 |
|---|---|---|
| B: 틱 사이즈 lookup table | 유형 2 | 크게 감소 |
| C: 과거 실수 예시 | 유형 1, 2, 4 | 적당히 감소 |
| D: "먼저 틱 사이즈 계산" CoT | 유형 2 | 크게 감소 |
| E: ALL | 모든 유형 | 가장 많이 감소 |

이 결과가 **"처방전"** 역할. 실무자가 "틱 LLM 전략 쓸 거면 이런 조치 하세요" 가이드로 사용.

### 4.5 왜 novel한가

교수님 질문 예상: *"failure taxonomy 논문이야 이미 많지 않나?"*

답: **LLM 코드 생성 failure 논문은 많지만 (HumanEval, SWE-bench 오류 분석) 마이크로구조 도메인은 0편.**

- 기존 failure taxonomy는 일반 프로그래밍 실수 (syntax error, logic error 등)
- 우리 유형들은 **도메인 특화**: 틱 사이즈 오류, 스프레드 해석 오류, OBI 게이트 누수 — 이런 건 어떤 LLM 평가 논문에도 없음

교수님 질문 예상 2: *"n=6 으로 taxonomy를 추출하는 건 통계력 부족 아닌가?"*

답: **동의. 이래서 실험 3 + corpus 확장이 중요.**
- 실험 3 (n=50)으로 각 intervention별 유형 발생률의 95% 신뢰구간 얻음
- 확장된 corpus에서 4유형 매트릭스 재확인

### 4.6 요약 — 기여점 3의 세 가지 새로움

1. **마이크로구조 도메인 특화 실패 분류** — 기존 LLM-코드 failure 논문에 없음
2. **각 유형에 대한 명확한 mitigation map** — 실무자 처방전
3. **"검사기 맹점"을 메타-기여로 제시** — "우리가 못 잡는 실패도 있다" 는 정직함이 오히려 future work 방향 제시

## 5. 교수님 관점에서의 질문 선제 답변

### Q1: "이거 그냥 negative result paper 아니야?"

**답**: 아니오. 두 가지 이유.

첫째, negative result (수익 안 남)는 **증거**이지 **주장**이 아님. 주장은 "수익 가능성이 horizon 의존적이다" 이고 이건 positive direction의 기여.

둘째, 우리는 **positive control (하루봉 샤프 1.12)** 을 확보함. 단순 "LLM이 트레이딩 못한다" 가 아니라 "**이 조건에서 잘하고, 저 조건에서 못한다. 이유는 X**" 를 보여주는 **구분적(differential) 결과**.

### Q2: "왜 트레이딩 도메인이어야 하나? 다른 도메인으로도 일반화되나?"

**답**: 이 방법론은 3가지 조건을 만족하는 모든 도메인에 일반화 가능.

조건 1. **결정론적 실행 환경**: 같은 입력 → 같은 출력
조건 2. **정량적 파라미터가 있는 스펙**: 임계값, 제한값 등이 명세에 명시됨
조건 3. **의미 있는 성능 지표**: 연속값 outcome

후보 도메인:
- SQL 쿼리 최적화 (스펙 = 쿼리 플랜, 파라미터 = timeout/memory limit)
- HPC 수치 커널 (스펙 = numerical precision 요구)
- 강화학습 정책 코드 (스펙 = reward structure)
- 분산 시스템 프로토콜 (스펙 = consistency level)

트레이딩은 **가장 깨끗한 testbed**: 결정론적(with seed), 정량 파라미터 많음, 성능 = PnL로 명확.

### Q3: "왜 지금인가? 3년 전에도 할 수 있지 않았나?"

**답**: 3년 전엔 불가능했고 지금만 가능한 5가지 요인.

1. **LLM이 실제로 trading 코드를 생성 가능해짐** (2023년 GPT-4 이후)
2. **AlphaForgeBench (KDD'26 in press)가 paradigm shift를 합의시킴** — "LLM이 action 아니라 code를 내라" 가 통념이 됨
3. **HFTBacktest 등 오픈소스 틱 엔진 성숙** (2024~25년)
4. **Binance public tick data 접근성** (2024~)
5. **Multi-agent 파이프라인 프레임워크 일반화** (LangGraph, Claude Agent SDK 등)

5개 요인의 교집합이 **2025년 말~2026년 초** 에 열렸음. 지금이 **정확히 adjacent possible 구간**.

### Q4: "n=6 pilot으로 taxonomy claim하는 건 너무 약하지 않나?"

**답**: 완전히 동의. **그래서 실험 1, 3으로 확장.**

- 현재 pilot: taxonomy가 **존재한다**는 existence proof (n=6)
- 실험 3 후: 각 유형의 **발생률** 정량 (n=50, 95% CI)
- 확장된 corpus 후: 유형 × symbol × LLM 세 축의 **이원 분산분석**

n=6 으로는 **현상 발견**, n=50 으로는 **정량화**, 이게 논문 내 논리적 단계.

### Q5: "venue 선택이 적절한가?"

**답**: NeurIPS D&B 트랙이 1순위, 근거:

- NeurIPS D&B는 **methodology + benchmark 이중 기여** 환영
- 우리 논문: 방법론 (측정 레이어) + 벤치마크 (horizon sweep corpus) 둘 다 제공
- 기존 LLM 트레이딩 논문이 KDD, ICAIF 같은 응용 venue 로 분산된 상태 → NeurIPS에 먼저 내면 더 넓은 AI 커뮤니티 주목 가능
- D&B는 reproducibility 가중 → 우리가 모든 코드/데이터/LLM version log 공개 → 적합

대안 venue: 
- KDD 2027 (AlphaForgeBench 접수 venue)
- COLM 2027 (agent framework 주목)
- MLSys (engineering 측면 강조)

### Q6: "이 논문이 받을 수 있는 가장 강한 비판은?"

**답**: 세 가지. 각각에 대한 우리 대응.

**비판 A**: "틱 LLM이 수익 안 나는 건 당연. 의미 없는 발견."
**대응**: "당연"한 이유를 구조적으로 분석했다는 게 기여. 임계값 $h^*$ 는 수수료의 함수로 quantify 됨. 또한 **어느 horizon에서 전환되는지는 당연하지 않음**.

**비판 B**: "Claude Sonnet 4.6 하나로만 측정하면 single-model bias."
**대응**: 실험 2에서 Opus 4.7, GPT-5.2 추가. Camera-ready 에서 해결.

**비판 C**: "tick 엔진이 custom이면 artifact일 수 있음."
**대응**: 
- 지금: synthetic HFTBacktest-style fill stream 으로 engine-agnostic POC (F7)
- Camera-ready: 실험 4로 live HFTBacktest data replication

세 비판 모두 이미 대응 방안이 논문 scope에 포함되어 있음 — robust한 구조.
