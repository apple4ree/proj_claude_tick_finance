4. 전형적 워크플로우 — 한 cycle #

Phase A

방향 정의

≤ 5분

Phase B

Iteration 실행

분 단위 (자동)

Phase C

결과 회고

5–15분
feedback-analyst 출력 + signal-improver의 ImprovementProposal 검토. 다음 iter 자동 제안 또는 Phase A로.
Phase D

코드 변경 시

간헐적
agent prompt 수정·schema 추가·reference 보강. 커밋 직전 /co-review로 Codex에 위임 검토. 설정은 /update-config.

Phase A·B·C는 같은 세션 안에서 반복. Phase D는 cycle 사이에 간헐적으로 발생하는 메타 작업.

6. Sub-agent 구성 (chain1) #

chain1/ 디렉토리 안의 8 agent. 각각 자기 폴더에 AGENTS.md, input_schema.py, output_schema.py, references/를 가짐.

Agent Stage 책임 주 input → output
signal-generator ① SignalSpec 후보 N개 생성. OBI/OFI/microstructure primitive whitelist 안에서, reference 인용 강제. GenerateInput → list[SignalSpec]
signal-evaluator ② SignalSpec 정적 검증 — formula 합법성, primitive whitelist 준수, no-lookahead, threshold range SignalSpec → EvaluationVerdict (pass/fail + reasons)
code-generator ②.5 SignalSpec → Python signal() 함수 합성. signal_fn_template.py 참조 강제. SignalSpec → strategy.py
fidelity-checker ②.75 SignalSpec ↔ 코드 AST-level 정합성. formula·threshold·horizon·entry rule·execution=1·no-lookahead 6항목 게이트. (SignalSpec, code) → FidelityVerdict
backtest-runner ③ 결정론적 실행 (LLM 추론 없음). signal()를 KRX/crypto LOB 데이터에 against, execution=1 strict. strategy.py + universe → BacktestResult
feedback-analyst ④ BacktestResult 분석. WR/expectancy/n_trades/regime별 분해. recommended_next_direction + retire 판정. BacktestResult → Feedback
signal-improver ⑤ (Spec, Result, Feedback) 배치 → ImprovementProposal. Atomic·single-axis 변경 강제. budget-aware. ImproveInput → list[ImprovementProposal]
chain2-gate gate 활성화 조건 (WR ≥ 55%, n_trades ≥ 50, ≥ 3일 재현) 평가. 통과 시 chain2 hand-off 태그. SignalSpec batch → Chain2GateVerdict
주의: backtest-runner는 의도적으로 LLM 추론을 갖지 않는 결정론적 실행자. "백테스트 결과를 LLM이 fudge하지 않게 한다"는 정책 (CLAUDE.md §🚫 #2 근거 기반 원칙)의 물리적 구현. 이 agent의 출력은 항상 같은 입력에 대해 같은 결과여야 함.

7.2 본문 6 섹션

섹션역할스타일
1. System Prompt 역할 1문장 + 절대 제약 4–6개 (불릿) "**Evidence-based**: every proposed spec must cite…" bold + 콜론 패턴.
2. User Prompt (template) 오케스트레이터가 채울 변수 슬롯 명시 {iteration_idx}, {prior_feedback_summary_or_none} 등. Code block 안에 plain text.
3. Reference 읽어야 할 파일 경로 + 한 줄 요약 오케스트레이터가 invocation 시 자동 로드. ../_shared/references/...로 상대 경로.
4. Input Schema Pydantic class 가리킴. 핵심 필드는 인라인 설명. input_schema.py:GenerateInput
5. Output Schema Pydantic class 가리킴. 강제 조건 명시. measured_* 필드는 None으로 emit, 등.
6. Reasoning Flow 번호 매긴 명령문 N단계. 마지막에 self-loop 명시. "If step N fails, do not emit — iterate from step M".

7.3 추가 관찰된 작성 패턴

  • Dated policy block — 정책 변경을 prompt 안에 인라인으로 기입. 예: **Diversity priority (Block C, 2026-04-21)**: explicitly seek specs that exercise primitives OUTSIDE the pure OBI/OFI family…
  • 식별자 서식 강제 — spec_id 같은 필드의 형식을 prompt에서 박음: iter{iteration_idx:03d}_{slug}
  • Self-check 의무 — Reasoning Flow의 끝 직전에 "Self-check against constraints" step. 5~6개 항목 (no execution logic / references present / primitives whitelisted / no lookahead / iteration_idx correct).
  • Helper vs Primitive 구분 — primitives_used에는 primitive 이름만 (obi_1, mid_px, ofi_proxy), helper(zscore, rolling_mean 등)는 formula string에서만 사용. prompt에 명시적 경고 박혀 있음.
예시: signal-generator의 Reasoning Flow 발췌 
1. Read references — summarize the key formula/claim from each reference you intend to cite.
2. Identify lesson — if iteration_idx > 0, read prior_feedback and prior_improvement;
                     extract ≥ 1 concrete lesson per spec.
3. Choose primitive(s) — select from whitelist; justify.
4. Compose formula — assemble; verify no lookahead; verify ops decompose into whitelisted ops.
5. Hypothesis statement — ≥ 20 chars, articulate why WR > 50%.
6. Self-check against constraints — confirm (a) no execution logic, (b) references present,
                                    (c) primitives whitelisted, (d) no lookahead, (e) iter idx ok.
7. Emit — serialize as SignalSpec JSON; assign spec_id = iter{idx:03d}_{slug}.

If step 6 fails, do not emit the spec — iterate from step 3.
예시: signal-improver의 Reasoning Flow 발췌 
1. Rank feedback batch — sort by (expectancy_bps desc, WR desc, n_trades desc).
2. Prune retires — drop any spec whose Feedback is `retire`.
3. Budget allocation — top spec 40%, second 30%, third 20%, tail 10%.
4. Generate mutations per seed — look up Feedback's recommended_next_direction in
                                  improvement_heuristics.md; apply recipe.
5. Avoid previously-tried — cross-reference prior_iterations_index.md;
                            if duplicate of RETIRED, pick next recipe.
6. Formalize — each proposal: parent_spec_id, proposed_mutations (atomic), search_axes.
7. Emit — list of ImprovementProposals, total ≤ budget.

객체역할핵심 필드
SignalSpec 한 신호 후보의 전체 명세 spec_id · name · hypothesis · formula · primitives_used · threshold · direction · prediction_horizon_ticks · references · iteration_idx · measured_WR · measured_expectancy

Feedback 한 (Spec, Result) 쌍에 대한 분석 결과 recommended_next_direction (enum) · retire (bool) · weakness 요약
