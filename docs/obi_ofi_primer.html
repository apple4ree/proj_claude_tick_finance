1. 왜 OBI / OFI를 다루는가

주가는 궁극적으로 매수/매도 주문의 흐름이 결정한다. 하지만 "누가 더 사고 싶어 하는가"를 직접 관찰할 수는 없고, 우리가 보는 건 거래소가 공개하는 호가창 (Limit Order Book, LOB)의 스냅샷뿐이다.

이 LOB 상태에서 "지금 시장이 어느 방향으로 기울어져 있는가"를 수치 하나로 요약하는 것이 OBI · OFI 계열 신호의 목적이다. 두 신호는 KRX · NYSE · Binance 등 다양한 시장에서 초 단위 이하 가격 방향을 예측하는 데 실증적 유의미성이 확인된 몇 안 되는 microstructure 신호다.

본 프로젝트의 Chain 1 전체가 "OBI · OFI 변형을 어떻게 조합해야 다음 tick의 방향 (up/down) 을 잘 맞히는가"를 자가 개선하는 파이프라인이다.

2. OBI — Order Book Imbalance (레벨 상태)

2.1. 직관

Bid(매수 호가) 쪽에 쌓여 있는 잔량과 Ask(매도 호가) 쪽에 쌓여 있는 잔량의 상대 비율. "누가 더 줄 서 있는가"를 0 ~ 1 범위에서 읽는다.

Bid-heavy 일수록 "사려는 사람이 더 많다 → 가격이 올라갈 확률이 높다"는 약한 통계적 신호가 된다 (뒤에서 반례 포함 해설).

2.2. 공식 (Top-of-book 1 level)

OBI1 = (Qbid1 − Qask1) / (Qbid1 + Qask1)
  • Qbid1 = 최우선 매수호가(BIDP1) 의 잔량
  • Qask1 = 최우선 매도호가(ASKP1) 의 잔량
  • 범위: [−1, +1]. +1 이면 ask 쪽 잔량 0 (사려는 사람만 있음), −1 이면 반대

2.3. 예시 스냅샷 (삼성전자 2026-03-26 09:00:16 실측)

Ask (매도)
LevelPriceQty
A10219,00024,622
A5216,5009,002
A1214,50017,800
Bid (매수)
LevelPriceQty
B1214,00069,659
B5212,000102,744
B10209,5008,556

여기서 OBI1 = (69,659 − 17,800) / (69,659 + 17,800) = +0.593. 매수 쪽이 매도의 약 4배 잔량이라 "단기적으로 위로 튀어오를 확률이 비대칭"이라는 해석.

2.4. 확장형

  • OBIk (k=3, 5, 10): 상위 k 레벨 누적 잔량 기준. 더 깊은 북까지 반영 → 스냅샷 노이즈에 강함
  • OBItotal: TOTAL_BIDP_RSQN / TOTAL_ASKP_RSQN 전체 잔량 비율
  • Microprice (Stoikov 2018): OBI1을 bps 가격 편차로 환산한 값. 수학적으로 microprice − mid = (spread/2) × OBI1 로 동등 (단, 스프레드 규모로 가중됨)
OBI의 성격: "지금 이 순간의 북 상태(state)"를 요약. slow-moving, noise 포함, 누가 뭘 했는지는 모름.

3. OFI — Order Flow Imbalance (흐름 변화)

3.1. 직관

OBI 가 "지금의 상태" 라면, OFI 는 "최근 tick 동안 어느 쪽으로 주문이 유입됐는가".
Bid 쪽에 주문이 새로 쌓이거나 Ask 쪽에서 주문이 취소되면 → "매수 압력 증가" 로 읽어 양수 OFI.

3.2. 공식 (Cont-Kukanov-Stoikov 2014 정식 정의, top-of-book)

매 스냅샷 t 마다 이전 스냅샷 t−1 과 비교:

bid_contrib =
   if Pbidt > Pbidt−1:   +Qbidt    (bid 가격이 올라왔다 = 새로운 높은 매수 등장)
   if Pbidt = Pbidt−1:   Qbidt − Qbidt−1    (같은 가격의 잔량 변화)
   if Pbidt < Pbidt−1:   −Qbidt−1    (bid 레벨이 빠졌다 = 매수 취소)

ask_contrib 도 대칭 (ask 가격이 내려오면 +, 올라가면 −)

OFI1 = bid_contrib − ask_contrib

"bid가 올라왔다" 와 "ask가 내려왔다" 는 둘 다 매수 우세 신호라 양수로 취합. 반대는 음수.

3.3. 변형

  • OFI proxy: KIS H0STASP0 에서 이미 제공하는 TOTAL_BIDP_RSQN_ICDC / TOTAL_ASKP_RSQN_ICDC 델타 필드를 이용.
    ofi_proxy = (Δbid − Δask) / (|Δbid| + |Δask|) → [−1, +1] 로 정규화된 근사치. 계산 비용 0.
  • Rolling OFI: rolling_mean(ofi, W=50) 같이 최근 W 틱 평균. 단일 tick 노이즈 제거용.
  • vol_flow: ΔACML_VOL, 즉 체결 물량 증가분. OFI 와 짝 짓는 공격적 체결 흐름 지표.

3.4. 왜 OFI가 OBI보다 예측력 강한가

OBI 는 과거 누적 + 현재 가 섞인 정적 상태. 하루 전부터 앉아 있던 잔량도 그대로 포함.
OFI 는 새로 도달한 정보 자체를 포착. 정보 유입이 → 가격 반영을 선행.

Cont-Kukanov-Stoikov (2014) 논문: NYSE 데이터에서 OFI가 10초 mid 변동 분산의 약 65%를 설명. OBI 단독으로는 약 30% 수준.

5. "신호가 있다"에서 "수익이 난다"까지 — 비용 벽 분해

"OBI1 > 0.2 이면 92% 확률로 오른다" 는 말이 수익을 보장하지 않는다는 것이 이 섹션의 핵심. 한 번의 왕복 거래 PnL 은 다음 방정식으로 결정된다:

net_pnl_per_rt = signal_edge  −  spread_cost  −  fees  −  taxes  ±  slippage  ±  adverse_selection

6. 실제로 돈을 버는 4가지 구조

6.1. MM 호가 skew — 가장 광범위

프롭 MM (Jump, Citadel Securities, Optiver, HRT, Jane Street 등) 은 매 tick 양쪽에 LIMIT 호가를 깔아두고 대기한다. 이때 OBI/OFI 는 호가를 어느 쪽으로 기울일지를 결정한다.

  • OFI 강한 + → ask 호가를 1 tick 올림 (팔기 덜 원함, 더 비싸게만 팜), bid는 그대로
  • OFI 강한 − → bid 호가를 1 tick 낮춤 (사기 덜 원함), ask는 그대로
수익원은 스프레드 × 거래량. OBI/OFI 는 불리한 체결(adverse selection) 회피 필터로 작동. 실제 PnL 기여는 "불리 체결 감소 → 평균 체결 수익 + 수십 bps-trades".

6.2. Multi-factor stat-arb의 한 축 (Two Sigma, DE Shaw 류)

Short-horizon 퀀트 펀드는 OFI 를 독립적 수익원으로 쓰지 않고, 수십~수백 개 factor의 ensemble 중 하나로 사용한다.

alphat = w1·OFI + w2·MomentumReversal_5m + w3·NewsSentiment + ... + wN·MicropriceDev
if alphat > threshold: enter

OFI 단독 0.4 bps edge 는 fee 벽을 못 넘지만, 여러 약한 신호의 가중합이 예: 10 bps 를 내면 fee 벽(8 bps)을 넘는다. "약한 신호 많이 vs 강한 신호 하나"의 수학적 해법.

6.3. Execution algo 타이밍 결정 (Buy-side 기관)

대형 기관이 몇 십만 주 매수해야 할 때 VWAP / TWAP / POV 알고리즘이 잘게 쪼개 집행한다. 이때 OFI 가 "언제 slice 를 내보낼지" 결정.

  • 사야 함 + OFI 강한 − → 잠깐 대기 (어차피 가격 내려올 예정)
  • 사야 함 + OFI 강한 + → 즉시 체결 (가격 올라가기 전)

수익이 아니라 비용 절감 형태의 PnL. Buy-side 는 "매수 집행 평균가 개선" 을 성과로 잡는다.

6.4. 극단치 event-driven taker (학술적으로 보고된 사례)

OFI 상위 1% 극단치 (top 0.1%) 시점에서는 raw edge 가 5 ~ 30 bps 로 점프한다 (Cont-Jain 2017 등). 이 구간은 하루 몇 번뿐이지만 크기가 커서 taker 로도 수익 가능.

  • 조건: 낮은 fee 시장 (선물 · 옵션 · 저수수료 ETF) + 빠른 latency
  • 거래 빈도: 하루 수 회 수준. 고빈도가 아니라 "선택적 고품질 신호 taker"
  • 본 프로젝트가 도달 가능한 가장 현실적인 형태 중 하나 (개인 레벨에서도 가능성 있음)

7.1. Chain 1의 역할

  • Whitelisted 프리미티브 (obi_1, obi_5, microprice_dev_bps, ofi_proxy, ofi_cks_1, vol_flow, rolling_*, zscore_* 등) 조합해서 SignalSpec 생성
  • 매 iteration 마다 signal-generator → signal-evaluator → code-generator → fidelity-checker → backtest-runner → feedback-analyst → signal-improver 루프
  • 목표: WR 55% 이상 · n_trades/day 50+ · 3일 이상 재현성 확보된 SignalSpec 을 최소 1개 산출 → Chain 2 게이트 통과

7.2. Chain 2의 역할 (미구현 · 설계만 완료)

  • Chain 1 이 넘겨준 SignalSpec 을 받아 ExecutionSpec 을 설계
  • hftbacktest 로 실제 체결 시뮬 (queue position, latency, multi-level fill, fee 실반영)
  • "이 신호 + 이 execution 조합에서 net PnL 이 실제로 양수인가?" 측정
  • 무수익 시 SignalSpec 을 Chain 1 에 execution_infeasible=true 태그와 함께 reject
왜 분리했나: WR 만으로 execution 을 자유롭게 움직이면 "tiny PT · huge SL" 류로 WR 을 100%에 수렴시킬 수 있지만 expectancy 는 음수. Execution 을 상수로 고정하면 WR 이 수학적으로 기대값을 결정 (E[return] ≈ (2·WR − 1) × E[|Δmid|]) 해서 조작 불가. Chain 1 의 WR 이 정직한 측정임을 보장하는 장치.
